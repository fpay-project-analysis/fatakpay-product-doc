1. Introduction — how the platform is organised

Every Fatakpay loan journey follows the same backbone: a customer logs in, a loan application is created or resumed, the user completes onboarding steps, credit checks run, an offer is proposed, the user accepts and sets up auto-debit, and finally money is disbursed into LMS (Loan Management System).

Fatakpay runs across three main codebases:

  • Onboarding — login, forms, KYC, CIBIL, offers, e-sign, NACH
  • Main backend — product configuration, NBFC masters, stage names
  • LMS — disbursed loan account, EMI schedule, repayments, card billing
Two product families:
product_master_id = 1 → Card (Fatak Cash) — revolving credit line products
product_master_id = 2 → EMI — term loans (Lite, Plus, Prime, Udaan)

Each application gets a unique code like FPA-00012345. The current step is stored as stage_id (see Section 3). Progress within form-filling is tracked by boolean flags on the application: pan, company, income, payment_flag, aadhar, and so on.

2. Login and application creation

What happens when a user opens the app

  1. User enters mobile number and verifies OTP.
  2. System finds or creates an AppUser record and issues an auth token.
  3. System checks whether an active LoanApplication already exists for this product (Card vs EMI).
  4. If no active application — or if the previous one was rejected/closed and re-application is allowed — a new application is created at Stage 1 (Form Filling).
  5. Dashboard decides which product banner to show (Card, EMI, or Udaan) based on eligibility flags.
When can a user start a fresh application?
Previous application at Stage 8 (Rejected) with is_able_to_new_application = True; or Stage 15 (Closed); or EMI fully repaid (check_if_emi_loan_closed).

Product channel (source code)

If the user arrived via a marketing link or partner, source_code maps to a ProductChannel. This can set product_master_id directly (Card or EMI) and enable direct onboarding — skipping the super-dashboard and landing straight on the PAN page.

Card vs EMI on the dashboard

If the user has both Card and EMI applications, the app opens whichever was updated most recently. Banner hints:

5. Card — One Month EMI (new user)

D2C Product 191 fatakcard_model

This is the primary Fatak Cash product for direct-to-consumer users. The customer applies for a revolving credit line that renews monthly. The journey has a distinctive soft offer path: after paying a platform fee, the user may see a dummy limit (Stage 38) before the full Decision Engine runs.

Step-by-step (plain English)

  1. Select Card — user chooses Fatak Cash on dashboard; application created at Stage 1.
  2. Form filling — PAN → company (B2B only) → income → platform fee → pre-approval step → Aadhaar → selfie → documents.
  3. CIBIL pull — bureau report fetched (CRIF). Stored in loan_application_cibildetails.
  4. Soft offer branch — if user paid platform fee, app moves to Stage 38 with a placeholder limit (~₹20k). If income step triggers early, fatakcard_model may run immediately.
  5. Full Decision Engine — fatakcard_model assigns a risk band R1 through R8.
  6. Band outcome:
    • R1–R7 → Approved. Limit typically ₹5,000 – ₹7,500 (band-dependent).
    • R8 → Rejected (Stage 8). User may see Udaan cross-sell.
    • API / data failure → Pre Approved (Stage 9) for manual follow-up.
  7. NBFC rules — product-NBFC mapping, pincode serviceability, name match.
  8. Offer Proposed (4) → acceptance → NACH → Disbursement (7). Card status set to Active.

Standardized BRE (fatakcash_new): Age 18–60, income ≥ ₹12,000, CRIF score ≥ 600, no 30+/60+/90+ DPD, no write-off. High-risk users (score > 750 or thin-file scores 12/14/15/16/17/18) also need 6-month bank statement, zero bounces, PAN-AA match.

6. Card — One Month EMI (repeat / monthly activation)

fatakcash_repeat_model

Existing card customers who already have a disbursed offer can activate a new monthly loan each cycle. This uses a separate repeat model — not the new-user fatakcard_model.

What the repeat model checks

  • DPD yesterday — Days Past Due on the previous cycle:
    • DPD ≤ 3 → eligible for CLI (credit limit increase)
    • DPD ≥ 4 (D2C) → CLD (limit decrease / block)
    • B2B users → CLD check skipped
  • Monthly loan record created in LMS for the new draw.
  • Cross-sell check (R1–R7) — if repeat band qualifies, user may be offered EMI product 290 (Lite reactivation).
Figure 4 — Repeat monthly activation flow
flowchart TD
    A[Existing disbursed card offer] --> B[User activates monthly loan]
    B --> C[fatakcash_repeat_model]
    C --> D{DPD yesterday?}
    D -->|≤ 3| E[CLI — limit increase]
    D -->|≥ 4 D2C| F[CLD — limit decrease / block]
    D -->|B2B| G[Skip CLD check]
    E --> H[Create monthly loan in LMS]
    H --> I{R1–R7 cross-sell band?}
    I -->|yes| J[Offer EMI 290 reactivation]
    I -->|no| K[Continue on card only]

7. Card — DCL (B2B partner)

B2B Black Box API

DCL (Debit Card Line) is a B2B card product where the credit limit comes from an external Black Box scoring API rather than the standard fatakcard model. Used for partner-led employee onboarding.

  1. B2B employee identified via partner_employeeuser.
  2. Black Box API returns a score → mapped to a credit limit.
  3. Offer Proposed with DCL sub-product.
  4. Convenience fee check — for D2C DCL, fee must be between ₹300–₹800. Outside range → Manual Disbursal (24).
  5. Disburse → KFS generated with DCL template.
Figure 5 — DCL B2B flow
flowchart TD
    A[B2B employee verified] --> B[Black Box API call]
    B --> C[Limit from score mapping]
    C --> D[Offer Proposed]
    D --> E{Convenience fee ₹300–800?}
    E -->|in range| F[Disburse]
    E -->|out of range| M[Manual Disbursal 24]
    F --> G[KFS — DCL agreement]

8. Card — EWA (Earned Wage Access)

B2B Corporate salary

EWA lets salaried employees access a portion of their earned wages before payday. Limit is calculated as corporate EWA percentage × monthly salary. Spending is gated by a daily attendance job.

  1. Corporate mapped via PartnerCorporatecardewamapping (record_type=EWA).
  2. Offer created with offer_type = EWA.
  3. Limit = configured percent × salary from income details.
  4. After disbursement, EWATransactionLimit tracks how much the user can spend.
  5. Daily attendance job updates amount_earned based on days worked.
  6. If attendance data missing → Pre Approved (9) until resolved.
Figure 6 — EWA flow
flowchart TD
    A[EWA corporate mapping] --> B[offer_type = EWA]
    B --> C[Limit = percent × salary]
    C --> D[Disburse card]
    D --> E[EWATransactionLimit set]
    E --> F[Daily attendance job]
    F --> G{amount_earned updated?}
    G -->|yes| H[User can spend earned amount only]
    G -->|no| P9[PRE-APPROVED Stage 9]

9. Card — D&C (Draw & Credit)

Existing users Multi-draw

D&C is a draw-down model: the user has a sanction pool (limit × tenure) and can take monthly draws from the remaining pool. Processing fee (PF) is charged once; AMC may renew. New D&C onboarding is largely inactive — this flow mainly serves existing users.

  1. Sanction amount = credit limit × tenure months.
  2. Remaining pool decreases with each child draw.
  3. If an active child loan exists → new draw blocked.
  4. PF charged once per parent offer; AMC on renewals.
  5. After 5 child loans → EMI cross-sell may trigger.
Figure 7 — D&C draw model
flowchart TD
    A[D and C parent offer] --> B["Sanction = limit × tenure"]
    B --> C[Remaining pool calculated]
    C --> D[User requests monthly draw]
    D --> E{Active child loan?}
    E -->|yes| X[Draw blocked]
    E -->|no| F[PF once + AMC on renewal]
    F --> G{≥ 5 child loans?}
    G -->|yes| H[EMI cross-sell offered]
    G -->|no| D

10. EMI — FatakLite

STPL Products 330, 343, 290 fatak_emi_non_banking_model

FatakLite is a short-term personal loan (STPL) for users who do not need bank statement / Account Aggregator verification. It uses the non-banking EMI model — bureau + income only.

Journey

  1. User starts EMI application (product_master_id = 2).
  2. Complete KYC: PAN, Aadhaar, selfie, documents. Failures → Pre Approved (9).
  3. CIBIL pull — CRIF bureau data processed.
  4. emi_non_banking_model assigns risk band. Band fail → Reject (8).
  5. Income check — declared income must be ≥ ₹15,000.
  6. CIBIL × Income matrix — determines loan amount and tenure.
  7. Offer Proposed → Accept → NACH → Disburse.
Lite variants:
• New users (330, 343) — CRIF ≥ 650, segment fataklite_new
• Reactivation (290) — CRIF ≥ 600, segment fataklite_disbursed_inactive
Figure 8 — FatakLite flow
flowchart TD
    A[EMI application started] --> B[CIBIL bureau pull]
    B --> C[fatak_emi_non_banking_model]
    C --> D{Risk band OK?}
    D -->|yes| E[Income ≥ ₹15,000]
    D -->|no| R8[REJECT Stage 8]
    E --> F[CIBIL × Income matrix]
    F --> G[Offer → NACH → Disburse]

11. EMI — FatakPlus

PL Products 276, 341, 336 fatak_emi_with_banking_model

FatakPlus is a full personal loan requiring bank verification via Account Aggregator (AA). The banking model (fatak_emi_with_banking_model) uses both bureau data and 6-month bank statement patterns.

Journey

  1. User completes form + KYC (same as Lite).
  2. Account Aggregator consent — user links bank account; transactions fetched.
  3. emi_with_banking_model runs rules R1, R2, R3:
    • Pass → loan amount and tenure from model output.
    • Fail → Reject (8).
  4. Product 336 (digital journey) — additional verified income check from banking data.
  5. NBFC mapping → Offer → NACH → Disburse.
Young user rule: If age < 25 and median banking income < ₹25,000 → rejected on Plus.
Figure 9 — FatakPlus flow
flowchart TD
    A[Products 276, 341, 336] --> B[Account Aggregator — bank link]
    B --> C[fatak_emi_with_banking_model]
    C --> D{Rules R1 R2 R3 pass?}
    D -->|yes| E[Amount + tenure from model]
    D -->|no| R8[REJECT Stage 8]
    E --> F{Product 336 — verified income?}
    F --> G[NBFC map → Disburse]

12. EMI — FatakPrime (DSA)

DSA Products 285, 221

FatakPrime serves the DSA (Direct Selling Agent) channel. Applications are identified by dsa_code on the loan application. Prime uses a dedicated engine first; if Prime fails, the system falls back to the Plus engine — giving the user a second chance at approval.

Journey

  1. DSA agent creates or assists application (TPP portal).
  2. CIBIL + bank data collected (AA or statement).
  3. Prime engine runs first:
    • Pass → Prime offer with FOIR caps applied.
    • Fail → Plus engine fallback.
  4. Plus fallback pass → offer as Plus terms; both paths lead to disbursement.
  5. Both fail → Reject (8).
Prime credit policy: Age 21–60, income ≥ ₹17,500 (no median income fallback for DSA), bureau score ≥ 650 or no-hit, banking: 6-month statement, up to 2 bounces, tiered AQB (₹1,000 or ₹2,000 if salary ≥ ₹50k).
Figure 10 — FatakPrime with Plus fallback
flowchart TD

13. EMI — FatakUdaan (Credit Builder)

Product 323 udaan_application_flag Program fee

FatakUdaan is a paid credit builder program for users who were rejected on Card/EMI or have CIBIL below 600. Unlike other EMI products, Udaan does not use a risk model — the loan amount and tenure come from admin configuration after the user pays a program fee.

Who sees Udaan?

  • Last non-Udaan application Rejected (Stage 8), OR
  • CIBIL score < 600 (and no active Udaan / no prior fee paid / no active LMS loan)

Full journey

  1. Udaan landing page — user views program; stage logged as "Form Filling - CIBIL".
  2. Pay program fee — Razorpay/Cashfree; on success → "Program Fee Paid" stage; invoice + welcome email sent.
  3. Auto-create application — payment callback creates LoanApplication with udaan_application_flag=True.
  4. CRIF soft pull — 3-stage CRIF API; credit dashboard shows tips (utilization, repayment, enquiries).
  5. PAN + name validation — income/occupation captured.
  6. KYC (Stage 32) — standard document flow.
  7. create_emi_offer — fixed offer from product config (Fatak Udaan, STPL). No model band check.
  8. Offer Proposed (4) → Accept (5) → NACH (6) → Disburse (7).
Figure 11 — FatakUdaan flow
flowchart TD
    A[Eligible: rejected OR CIBIL < 600] --> B[Pay program fee]
    B --> C[CRIF soft pull + PAN + name check]
    C --> D[Admin-configured offer — no risk model]
    D --> E[Offer Stage 4 → Disburse Stage 7]

14. Card reject → Udaan cross-sell

When a Card Decision Engine returns Reject (Stage 8), the app checks udaan_eligible_flag. If true, the user sees a Udaan entry banner instead of a dead end.

Figure 12 — Reject to Udaan cross-sell
flowchart LR
    DE[Card Decision Engine] -->|REJECT Stage 8| F{show_udaan eligible?}
    F -->|yes| U[Udaan landing + program fee]
    F -->|no| END[Journey ends]
    U --> FEE[Program fee payment]
    FEE --> OFF[Config STPL offer → Disburse]

The API returns show_udaan_flag: true on reject responses when eligibility passes. User-level flag show_udaan_banner on AppUser can also control visibility.
